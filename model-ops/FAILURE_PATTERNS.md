# FAILURE_PATTERNS.md — 자주 망치는 패턴과 수정법
> 사용법: 작업 시작 전에 훑고, 검수 때 이 목록을 체크리스트로 돌린다.
> 각 패턴: 원인 → 증상(발견법) → 수정법.

## F1. 글씨 과다
- **원인**: 설명으로 신뢰를 만들려는 습관. 섹션마다 소개 문단을 붙임.
- **증상**: 한 화면에 문장 5개 이상. 스크롤 두 화면 연속 이미지 없음. 헤드라인이 두 줄 넘음.
- **수정법**: 섹션 본문 ≤3문장으로 자르고, 나열 정보는 리스트/스텝으로, 법적·상세 설명은 접기(더보기)로. 헤드라인 ≤15자로 재작성. 지운 문장은 잃는 게 아니라 하위 페이지로 옮긴다.

## F2. 이미지가 작음 / 장식용
- **원인**: 텍스트를 주인공으로 두고 이미지를 곁들임으로 처리.
- **증상**: 히어로 이미지가 화면 폭의 70% 미만, 카드 썸네일이 100px 미만, 아이콘 나열로 이미지를 대체.
- **수정법**: 히어로는 폭 100% × 뷰포트 높이 40~60%. 상품/사례 카드는 이미지가 카드 면적의 60% 이상. 실물 사진이 없으면 사진 요청을 보고에 명시하고 임시로라도 실물 계열 사진을 쓴다(추상 일러스트 대체 금지).

## F3. CTA 약함
- **원인**: CTA를 여러 개 만들어 서로 경쟁시키거나, 폴드 아래에 배치.
- **증상**: 첫 화면 스크린샷에 버튼이 없음. 같은 화면에 "문의하기/상담하기/신청하기"가 다른 스타일로 공존. 라벨이 "더 알아보기".
- **수정법**: 1차 CTA 1종류로 통합, 폴드 위 + 페이지 끝 재노출. 라벨은 행동+대상. 나머지는 2차(보더 버튼)나 텍스트 링크로 강등.

## F4. 모바일 폴드(첫 화면) 실패
- **원인**: PC 화면부터 만들고 모바일로 줄임. 히어로에 공지/배너/인사말을 먼저 배치.
- **증상**: 375×812 스크롤 0 스크린샷에서 CTA 또는 핵심 이미지가 안 보임.
- **수정법**: 375px에서 먼저 설계. 폴드 위 구성은 4요소(로고/헤드라인/이미지·숫자/CTA)로 고정하고 나머지는 전부 아래로 내린다.

## F5. 무의미한 그라데이션 / 과한 효과
- **원인**: "고급스럽게"를 효과로 해석. AI 기본 출력 스타일(보라-파랑 그라데이션, 글래스모피즘, 큰 그림자).
- **증상**: 배경에 2색 그라데이션, 카드에 blur/유리 효과, 그림자 20px 이상, hover에 요소가 떠오르는 애니메이션 남발.
- **수정법**: 배경은 단색(#fff/#f4f1ea 계열). 그라데이션은 사진 위 텍스트 가독용 어두운 오버레이만. 그림자는 0~`0 1px 3px rgba(0,0,0,.06)`. 고급감은 효과가 아니라 여백+큰 실물 사진+규격 일관성으로 만든다.

## F6. 템플릿 느낌
- **원인**: 범용 랜딩 패턴(중앙 정렬 아이콘 3열 + 제목 + 설명)을 반복 조립.
- **증상**: 아이콘 3열 섹션이 2개 이상. 어느 업종에 갖다 놔도 말이 되는 화면. 이모지/AI 아이콘 사용.
- **수정법**: 아이콘 섹션은 페이지당 최대 1개. 업종 고유 콘텐츠(실제 시계 사진, 실제 정비 사례, 실제 절차)로 교체. "이 섹션이 경쟁사 사이트에 있어도 어색하지 않은가?"를 물어서 그렇다면 다시 만든다.

## F7. 섹션 반복 / 목적 없는 섹션
- **원인**: 분량을 채우려고 비슷한 섹션을 늘림.
- **증상**: 같은 레이아웃 패턴 2회 이상. "회사 소개"성 섹션이 흩어져 3곳. 섹션 목적을 물으면 답이 없음.
- **수정법**: 섹션마다 목적(전환/신뢰/정보/탐색) 1개를 적고, 목적이 겹치는 섹션은 합친다. 랜딩은 5~7섹션으로 제한.

## F8. 브랜드 감도 부족 (업종 톤 미스매치)
- **원인**: 업종 무관 중립 톤으로 제작.
- **증상**: 명품시계 커머스가 스타트업 SaaS처럼 보임. 정비센터가 동네 광고 전단처럼 보임. 원색 남발, 유치한 카피.
- **수정법**: 업종 기준 적용 — 벨로르=바이버/구구스식 상품 사진 중심+절제된 색, 프로모터스=어두운 톤 실제 작업장/차량 사진+정돈된 예약 동선, 오토픽스=절차와 숫자 중심의 신뢰 화면. 레퍼런스 캡처가 있으면 그대로 따른다.

## F9. 반응형 깨짐
- **원인**: 한 폭에서만 확인하고 제출. flex/grid 기본값 함정.
- **증상**: 특정 폭에서 텍스트 잘림, 버튼 줄바꿈, 가로스크롤, 요소 겹침.
- **수정법**: 375/768/1280 3개 폭 확인을 제출 조건으로. 알려진 원인 우선 점검: flex 자식 `min-width:auto`(→ `min-width:0`), 버튼 `white-space:nowrap` 누락, 고정폭 px 하드코딩(→ max-width+%), 절대배치 요소의 폭 미대응.

## F10. 과장·보장성 카피
- **원인**: 전환 욕심으로 확언을 씀.
- **증상**: "업계 최고가 보장", "100% 만족", "무조건", "최저가".
- **수정법**: 검증 가능한 사실로 교체 — "구성품·상태 확인 후 확정 견적", "전국 출장 방문", "당일 정산". 확정 정책(예: 가품 판명 시 전액환불)만 확언 허용.

## F11. 겉만 수정 (연동 누락)
- **원인**: 지적받은 화면만 고치고 데이터 흐름 전체를 안 따라감.
- **증상**: 폼은 바뀌었는데 저장 payload/DB 컬럼/상세 표시/서버 검증은 옛 필드. 같은 값이 두 파일에 다르게 하드코딩.
- **수정법**: 필드 하나를 바꾸면 그 필드를 grep해서 지나가는 모든 지점(입력→payload→DB→표시→서버)을 나열하고 전부 수정. 수정 지점 목록을 보고에 포함.

## F12. 오버레이 z-index 충돌
- **원인**: 새 모달/토스트에 임의의 고정 z-index를 부여.
- **증상**: 상위 패널 위에서 새 창이 뒤에 깔림, 닫기 불가.
- **수정법**: 새 오버레이 추가 전 기존 최상위 z-index를 확인하고 그보다 위로. 전역 감시(MutationObserver로 body 전체 감시+감시 대상 수정)는 금지 — 요소별 개별 처리.

## F13. 미확인 배포
- **원인**: "코드상 맞을 것"이라는 추측으로 배포.
- **증상**: 배포 후 스플래시 멈춤·화면 백지 등 치명 장애.
- **수정법**: 전역/대규모 변경은 실제 구동 확인 전 배포 금지. 실구동이 불가한 환경이면 격리 렌더로라도 확인하고, 그마저 못 했으면 보고에 "실구동 미확인"을 명시해 사장님이 확인 후 배포하게 한다.

## F14. 플레이스홀더 잔존
- **원인**: 더미로 채우고 교체를 잊음.
- **증상**: lorem ipsum, "제목을 입력하세요", 깨진 이미지 아이콘, # 링크.
- **수정법**: 더미가 필요하면 실제 사용 가능한 한국어 문안·실물 계열 이미지를 넣는다. 제출 전 "lorem", "TODO", "입력하세요", `href="#"` 검색을 검수 루틴에 포함.
